在软件开发过程中，我们经常需要撤销一些不想要或者错误的更改。Git 提供了多种强大的命令来回滚代码。本文主要介绍如何将工作区的代码恢复到上一次提交（HEAD）的状态。

### 场景分析

当你做了一些修改，但发现这些修改是错误的或者暂时不需要时，你可能希望“一键撤销”所有改动，让工作目录回到最后一次提交时的干净状态。

根据你的修改所处的阶段（仅在工作区、已暂存或已提交），回滚的方法有所不同。

### 1. 撤销工作区的修改（未 `git add`）

这是最简单的情况。你修改了一些文件，但还没有将它们添加到暂存区。

**命令：**

```bash
git checkout -- .
```

或者，对于较新版本的 Git，推荐使用 `restore` 命令，它的语义更清晰：

```bash
git restore .
```

**作用：**
这两个命令都会丢弃工作区中所有已跟踪文件的修改，将它们恢复到 `HEAD`（上一次提交）的状态。`.` 表示当前目录下的所有文件。

**警告：** 这是一个危险的操作，因为所有未保存的本地修改都会丢失，且无法恢复。

### 2. 撤销已暂存的修改（已 `git add`）

如果你已经执行了 `git add`，但还没有 `git commit`，你需要分两步操作：

**第一步：取消暂存（Unstage）**

使用 `git reset` 将文件从暂存区移回到工作区：

```bash
git reset
```

或者，使用 `restore` 命令：

```bash
git restore --staged .
```

执行后，`git status` 会显示这些文件从 "Changes to be committed" 变回了 "Changes not staged for commit"。

**第二步：撤销工作区的修改**

现在情况回到了第一种场景，你可以使用 `checkout` 或 `restore` 来丢弃工作区的修改：

```bash
git restore .
```

### 3. 撤销已提交的修改（已 `git commit`）

如果你已经错误地提交了代码，想完全撤销这次提交，可以使用 `git reset`。

**命令：**

```bash
git reset --hard HEAD~1
```

**作用：**
这个命令会将你的分支指针（例如 `main`）和 `HEAD` 一起移动到上一次提交之前的那个提交（`HEAD~1`）。

-   `--hard` 参数会彻底丢弃工作区和暂存区中由这次错误提交引入的所有更改。
-   `HEAD~1` 指的是当前提交的父提交。

**警告：**
`git reset --hard` 是一个非常强大的命令。如果你的提交已经推送到了远程仓库，**强烈不推荐**使用它，因为它会改写公共历史，给其他协作者带来麻烦。在这种情况下，更安全的做法是使用 `git revert`，它会创建一个新的提交来撤销指定提交的更改。

### 总结

| 状态 | 命令 | 作用 |
| :--- | :--- | :--- |
| **仅在工作区** | `git restore .` | 丢弃所有未暂存的修改。 |
| **已暂存** | 1. `git restore --staged .`<br>2. `git restore .` | 先取消暂存，再丢弃修改。 |
| **已提交 (本地)** | `git reset --hard HEAD~1` | 彻底移除最后一次提交及其更改。 |

选择哪种回滚方式取决于你的更改进行到了哪一步。请务必谨慎操作，尤其是在处理已提交或已推送的更改时。
